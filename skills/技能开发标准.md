# dimens-cli/skills 技能开发标准

本文档基于 `skill-creator` 的技能设计原则，并结合维表智联当前 3 个顶层 Skill 体系制定。

## 1. 当前技能架构

当前只允许发布 3 个顶层技能：

| 顶层 Skill | 职责 | 适用场景 |
| --- | --- | --- |
| `dimens-system-orchestrator` | 系统级拆解、执行顺序、章节路由 | 用户要生成完整业务系统、平台、管理应用 |
| `dimens-manager` | 项目内业务管理 | Key、团队、项目、表格、权限、工作流、报表等业务落地 |
| `dimens-sdk` | 开发者集成 | SDK、HTTP API、Web、BFF、Node.js、移动端接入 |

新增能力时，默认不要新增顶层 Skill，优先判断是否应该作为：

- `dimens-manager/references/<业务域>/` 下的新章节或补充文档
- `dimens-system-orchestrator/references/` 下的系统级路由或案例
- `dimens-sdk/references/` 下的接入示例或 API 说明

## 2. 什么时候可以新增顶层 Skill

只有同时满足下面条件，才考虑新增顶层 Skill：

1. 它不是维表智联项目内业务管理的一部分。
2. 它不是 SDK / HTTP / Web / BFF / Node.js 接入的一部分。
3. 它不是系统拆解、执行顺序或路由案例的一部分。
4. 它有独立触发场景，和现有 3 个顶层 Skill 不会互相抢入口。
5. 它的 `description` 能明确说明“什么时候用”，而不是描述“它做了什么”。

如果只是新增一个业务域，比如应用市场、消息、知识库、模型配置，默认应先进入 `dimens-manager/references/<业务域>/`，而不是新增顶层 Skill。

## 3. SKILL.md 标准

每个顶层 Skill 必须包含 `SKILL.md`，并满足：

```yaml
---
name: dimens-example
description: 用于维表智联某类具体触发场景，说明什么时候应该使用该技能。
---
```

### 3.1 frontmatter 规则

| 字段 | 要求 |
| --- | --- |
| `name` | 只使用英文、数字、连字符，必须和目录名一致 |
| `description` | 只写触发条件，不写流程总结；控制在 500 字以内 |
| `slug` | 可选，但建议和 `name` 一致 |
| `version` | 可选，当前统一 `1.0.0` |
| `author` | 当前统一 `方块智联工作室` |
| `tags` | 用于平台检索，保持少而准 |

### 3.2 description 写法

推荐：

```yaml
description: 用于维表智联系统级方案拆解与执行编排，适合“生成一个客户管理系统/平台”这类完整业务系统搭建需求。
```

不推荐：

```yaml
description: 先分析需求，再拆模块，再路由到 manager，然后执行命令。
```

原因：`description` 是技能发现入口，只应该回答“什么时候用”，不要把流程写进去，避免 AI 只看描述就跳过正文。

## 4. SKILL.md 正文结构

顶层 `SKILL.md` 推荐使用下面结构：

```markdown
# 技能名称

## 执行前必读

## 职责边界

## 快速路由表

## 默认处理顺序

## 高风险跑偏点

## 常见错误与修正

## 参考文档
```

### 4.1 正文原则

- 保持短：`SKILL.md` 只放核心规则、路由表和高风险约束。
- 细节下沉：接口、命令、案例、矩阵、复杂说明放入 `references/`。
- 不重复：同一条规则不要同时在 `SKILL.md` 和多个 references 里重复维护。
- 可执行：涉及 CLI 的地方必须给出命令入口或明确指向命令文档。
- 可验证：新增或修改命令口径时，必须同步补测试或至少补 dry-run 验证。

## 5. references 标准

### 5.1 适合放入 references 的内容

| 类型 | 示例 |
| --- | --- |
| 接口说明 | API 路径、请求体、响应体、字段含义 |
| 命令映射 | CLI 命令、参数、返回结构、注意事项 |
| 场景案例 | 客户管理系统、售后工单、报表生成、权限隔离 |
| 能力状态 | 已封装、仅文档覆盖、当前不支持 |
| 复杂规则 | 权限矩阵、字段规则、报表组件映射、模型路由 |

### 5.2 命名规则

推荐文件名：

- `overview.md`：章节总览
- `examples.md`：案例
- `command-mapping.md`：命令映射
- `capability-status.md`：能力状态
- `usage.md`：使用分层
- `scenario-routing.md`：场景路由

不推荐：

- `最终版.md`
- `新文档.md`
- `接口说明2.md`
- `临时整理.md`

### 5.3 manager 章节标准

`dimens-manager` 内部业务域统一结构：

```text
dimens-manager/references/<domain>/
├── README.md
├── overview.md
├── references/
├── rules/
└── assets/
```

注意：业务域入口必须叫 `overview.md`，不能叫 `SKILL.md`，否则平台可能误识别成顶层技能。

## 6. scripts 标准

只有当操作需要确定性、重复执行或容易写错时，才新增 `scripts/`。

适合放脚本：

- 批量校验 Skill 路径
- 生成发布清单
- 检查 `SKILL.md` 数量
- 校验 references 断链

不适合放脚本：

- 一次性文本整理
- 可以用简单命令完成的检查
- 需要频繁人工判断的业务设计

## 7. assets 标准

每个顶层技能允许有：

```text
assets/
└── README.md
```

如需正式素材，统一命名：

- `cover.png`
- `icon.png`
- `screenshot-01.png`
- `screenshot-02.png`
- `screenshot-03.png`

具体规范看 `assets-命名规范.md`。

## 8. 技能开发流程

新增或改动 Skill 时，按下面顺序：

1. 判断目标属于哪个顶层 Skill。
2. 判断是改 `SKILL.md`，还是改 references。
3. 搜索现有文档，避免重复造章节。
4. 先写最小必要内容，不堆大段背景。
5. 如果涉及 CLI 命令，查真实代码和测试。
6. 如果涉及接口响应，必须以当前实现为准。
7. 更新根 README 或相关路由文档。
8. 做路径和打包验证。

## 9. 必做验证

发布前建议至少执行：

```bash
find dimens-cli/skills -name SKILL.md -print | sort
```

期望只看到：

```text
dimens-cli/skills/dimens-manager/SKILL.md
dimens-cli/skills/dimens-sdk/SKILL.md
dimens-cli/skills/dimens-system-orchestrator/SKILL.md
```

再执行：

```bash
cd dimens-cli
npm pack --dry-run --json
```

确认：

- 包内只有 3 个顶层 `SKILL.md`
- 没有过期检查清单
- 新增 references 文件进入发布包

如果改了 CLI 代码，还要按范围执行：

```bash
pnpm test <相关测试文件>
pnpm typecheck
```

## 10. 高风险错误

| 错误 | 后果 | 修正 |
| --- | --- | --- |
| 新业务域直接新增顶层 Skill | 顶层入口变多，AI 路由噪音上升 | 放到 `dimens-manager/references/<domain>/` |
| manager 业务域里放 `SKILL.md` | 平台误识别为独立技能 | 改名为 `overview.md` |
| description 写成流程总结 | AI 可能跳过正文直接执行 | 改成触发条件 |
| 文档只按旧接口写 | 技能执行会报错 | 对照当前代码和真实接口返回 |
| 只改文档不改测试 | 技能与 CLI 行为继续漂移 | 涉及命令必须补测试 |
| 旧技能名继续作为路由目标 | 误导使用者以为还有多个顶层 Skill | 改为 dimens-manager/references/{domain}/overview.md |

## 11. 当前报表案例教训

报表创建曾出现 `Cannot read properties of undefined (reading 'reportId')`，根因是文档和技能仍按旧 `/app/report/:projectId/add` 理解创建返回，但当前真实产品创建项目菜单报表走：

```text
/app/mul/project/:projectId/sheet/create
```

返回里主要是：

```text
sheetId = sh_xxx
```

服务端实际约定是：

```text
reportId = sheetId
```

因此技能标准要求：凡是涉及接口返回结构，必须对照当前实现和真实响应；如果 CLI 需要继续衔接下一步命令，应在 SDK 或命令层做返回结构归一化。
