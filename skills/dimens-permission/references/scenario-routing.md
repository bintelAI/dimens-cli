# dimens-permission 场景排查路径

## 1. 文档目标

这份文档专门解决一个问题：

权限问题很容易知道“五层结构”，但用户真正问出来的通常是场景句子，而不是层级句子。

例如：

- 能进项目但看不到表
- 行分页正常，但协同还是越权
- 公开角色明明只能看自己，却还能看到别人

这份文档负责把这些场景直接路由到正确的权限层和接口层。

---

## 2. 场景 1：能进项目但看不到表

### 标准判断

1. 先确认项目准入是否通过。
2. 再确认项目综合权限快照。
3. 再确认表级权限是否继续收窄。

### 解释模板

```md
能进入项目，只代表通过了准入层。

如果看不到表，下一步要继续看项目综合权限快照和表级权限，而不是把“准入通过”误判成“资源授权通过”。
```

### 推荐落点

- `examples.md`：看项目准入和项目权限接口
- `matrix.md`：看准入层、表级层之间的关系

---

## 3. 场景 2：前端可编辑，提交后被拒绝

### 标准判断

1. 前端是否只是做了保守预判。
2. 后端最终权限服务是否拒绝。
3. 被拒绝的是表级、列级还是行级。

### 解释模板

```md
前端按钮可点只代表前端预判允许尝试，不代表后端一定放行。

最终要以后端权限服务为准，并继续落到表级 / 列级 / 行级哪一层拒绝。
```

### 推荐落点

- `matrix.md`：看列级层、行级层
- `examples.md`：看 `row_policy/check` 与项目权限相关接口

---

## 4. 场景 3：行分页正常，但协同还是越权

### 标准判断

1. 普通读取链路是否正常。
2. 协同权限快照缓存是否独立隔离。
3. 广播过滤是否按接收者逐个裁决。
4. 系统字段是否先净化再鉴权。

### 解释模板

```md
行分页正常，只能说明 HTTP 读取链路基本正常。

协同链路还要继续检查权限快照、系统字段净化和广播过滤，所以这两条链路不能直接画等号。
```

### 推荐落点

- `matrix.md`：看协同投影层
- `examples.md`：看 Yjs 协同入口与权限快照相关接口

---

## 5. 场景 4：公开角色配置成“只能看自己”，为什么还能看到别人

### 标准判断

1. 当前用户是不是公开访问者。
2. 项目公开角色映射是否正确。
3. 行级策略是否真的按当前上下文生效。
4. 协同广播是否错误复用了更宽权限快照。

### 解释模板

```md
这类问题不能只看公开角色配置本身。

还要继续检查：
1. 公开访问者身份是否成立
2. 行级策略是否按当前上下文真正命中
3. 协同广播有没有按接收者做过滤
```

### 推荐落点

- `examples.md`：看项目准入、公开访问者、行级策略接口
- `matrix.md`：看准入层、行级层、协同层

---

## 6. 场景 5：同一用户在不同项目权限不一致

### 标准判断

1. 是否同属一个团队。
2. 项目准入是否一致。
3. 项目级角色、表级角色、部门挂载关系是否一致。
4. 是否存在不同的公开角色或数据访问范围。

### 解释模板

```md
同一用户在不同项目权限不同是正常现象，不能直接按“同一个账号”推导权限一致。

必须按项目逐个检查准入、项目角色、表级角色和行级策略。
```

## 7. 与其他 references 的关系

建议按下面顺序组合使用：

1. 先看本文件，把用户问题路由到正确场景。
2. 再看 `matrix.md`，确认这类问题属于五层里的哪几层。
3. 最后看 `examples.md`，确认真实接口和入参出参。
