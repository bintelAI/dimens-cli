# dimens-team 隔离与角色边界

## 1. 基本原则

团队是最高隔离边界，项目是团队下的业务单元。

因此：

- 任何团队外的数据默认不可见
- 任何项目级能力都默认从团队上下文进入

这份文档只聚焦：

1. 团队与项目的隔离边界
2. 团队级 / 项目级角色的判断

上下文来源请看 `context-sources.md`。

项目入口和系统建设上游意义请看 `project-entry.md`。

## 2. 资源隔离边界

默认要先区分两层资源：

| 层级 | 含义 | 典型问题 |
| --- | --- | --- |
| 团队级 | 最高隔离边界 | 用户是否属于该团队 |
| 项目级 | 团队下的业务单元 | 用户是否能进入该项目 |

因此分析“看不到资源”时，不能只看项目成员关系，必须先看团队归属。

## 3. 角色的两层结构

### 3.1 团队级角色

团队级角色主要影响：

- 成员管理
- 部门管理
- 团队级设置
- 团队公开资源可见性

### 3.2 项目级角色

项目级角色主要影响：

- 项目成员管理
- 项目内容编辑
- 项目只读访问

## 4. 为什么不能只看项目成员表

常见误区是：

- 用户没在项目成员表里，就默认判断为看不到项目
- 用户在项目成员表里，就默认判断为一定可管理项目

这两个结论都不稳定，因为还可能叠加：

- 团队角色
- 部门主管角色
- 项目公开可见性

## 5. 为什么这个 Skill 是其他 Skill 的上游

表格、文档、工作流、报表、权限等 Skill 都会依赖 `teamId` 和 `projectId`。

因此当其他 Skill 无法明确上下文时，应该先回到 `dimens-team` 的判断逻辑。
