# 工作流总规范

本文档定义 `dimens-manager` 工作流章节的共用规则，供 `AI 工作流`、`审批工作流` 和后续其他场景模板复用。

## 1. 适用范围

工作流章节统一覆盖以下内容：

1. 工作流总入口与章节路由
2. 节点词典与节点职责
3. AI 工作流生成模板
4. 审批工作流生成模板
5. 项目挂载、运行调用、模型边界与验证口径

## 2. 输出契约

所有工作流生成输出都应至少包含以下四部分：

1. 业务目标或场景说明
2. 节点列表
3. 节点之间的连线
4. 项目落地或调用说明

如果是结构化草案，还应包含：

- `pluginType`
- `nodes`
- `edges`
- `globalVariables`
- `meta`

## 3. 节点分层

### 3.1 通用分层

工作流节点默认分成五层：

1. 开始与输入层
2. 处理与判断层
3. 人工或模型执行层
4. 通知、回写、归档层
5. 结束层

### 3.2 生成时的基本原则

- 每个节点只表达一个清晰动作。
- 条件节点必须能清楚表达分支含义。
- 人工审批或人工确认节点必须能定位到具体处理人策略。
- 结束节点必须能收口，不要把所有路径混到一个出口。

## 4. 连线规则

1. `source` 和 `target` 必须指向已存在节点。
2. 条件节点至少要有两条可解释的出边。
3. 通过、拒绝、回写、撤回、超时等结果路径要尽量分开。
4. 不要让孤立节点悬挂在图中。

## 5. 变量规则

1. `globalVariables` 只放跨节点共享的核心变量。
2. 变量名使用稳定、可读的英文短名。
3. 变量定义必须能支撑节点执行，不要把整张表字段无脑塞进全局变量。

## 6. 校验规则

生成或解释工作流时，至少检查以下内容：

1. 节点 `id` 是否唯一。
2. 每条边是否引用有效节点。
3. 开始节点是否能到达主要路径。
4. 是否至少能到达一个结束节点。
5. 人工节点是否有明确的审批人策略。
6. 条件节点是否有清晰分支语义。
7. 每个 `node.type` 是否来自当前实现或对应场景白名单；不能为了表达语义编造新类型。
8. 审批工作流发布接口下，禁止输出 `action`、`sync_workflow_cell`、`approval_ai_review` 这类不支持或已废弃类型。

## 7. AI 工作流与审批工作流的共用边界

AI 工作流偏向：

- 任务编排
- 模型调用
- 工具调用
- 结果汇总

审批工作流偏向：

- 人工审批
- 条件分支
- 任务挂起与恢复
- 审批结果回写

两者可以共享节点词典，但不能混淆业务语义。
