# dimens-manager 多维表格章节 字段设计模式

## 1. 文档目标

这份文档专门回答两个问题：

1. 一张业务表应该怎样把字段设计到可落地、可筛选、可排序、可关联。
2. relation 字段到底要补齐哪些配置，后续行数据查询才不会难用。

补充一条：

3. 哪些字段设计会直接影响后续报表的 `preview`、`dataMapping` 和图表成功率。

它不代替接口清单，接口方法、路径、入参、出参仍以 `examples.md` 为准。

---

## 2. 字段设计快速索引

设计一张表时，建议至少先写清下面这些字段能力：

| 设计项 | 是否建议默认提供 | 说明 |
| --- | --- | --- |
| 字段名 | 是 | 用户能直接理解的业务名称 |
| 类型 | 是 | `text` / `number` / `date` / `select` / `relation` 等 |
| 是否主展示字段 | 是 | 列表标题、relation 展示默认优先读它 |
| 是否必填 | 是 | 是否允许为空 |
| 是否唯一 | 建议 | 编号、手机号、合同号等通常需要 |
| 默认值 | 建议 | 状态、负责人、日期等常见有默认值 |
| 是否参与搜索 | 建议 | 是否纳入 `keyword + searchFieldIds` |
| 是否参与筛选 | 是 | 是否适合 `filters` |
| 是否参与排序 | 建议 | 是否适合 `sortRule` |
| 备注 | 建议 | 字段用途、值约束、是否需要枚举 |

如果是 relation 字段，还要继续补齐：

| relation 设计项 | 是否建议默认提供 | 说明 |
| --- | --- | --- |
| 目标表 `targetSheetId` | 是 | 关联到哪张表 |
| 展示字段 `displayColumnId` | 是 | 关联值最终显示哪个字段 |
| 是否多选 | 是 | 一对一还是一对多 |
| 是否双向 | 建议 | 是否需要反向关系 |
| 编辑视图字段 `editViewFields` | 是 | 关联选择器里展示 / 搜索哪些字段 |

如果是 `select` 或 `multiSelect` 字段，还要继续补齐：

| 枚举设计项 | 是否建议默认提供 | 说明 |
| --- | --- | --- |
| 候选项 `options` | 是 | 例如 `潜客 / 跟进中 / 成交 / 流失` |
| 数据源类型 | 是 | 自定义场景固定用 `manual` |
| 默认值 | 建议 | 必须来自候选项之一 |
| 是否允许后续扩展 | 建议 | 先确认是否稳定枚举 |

补充判断：

- 如果字段看起来像“负责人、处理人、成员、审批人、跟进人、归属人”这类人员选择，而且项目本身已经有自己的用户体系、部门体系和内置角色，这通常不是普通 `select`，而应优先设计为 `person`
- 如果字段看起来像“所属部门、负责部门、归属组织、发起部门”这类组织选择，这通常不是普通 `select`，而应优先设计为 `department`
- `select` / `multiSelect` 只适合稳定静态枚举；人员、部门这类组织型字段属于特殊情况，不要混用

## 2.1 与报表联动的字段设计原则

如果业务表后续还会进入报表，字段设计时默认要先区分三类：

| 字段角色 | 推荐类型 | 典型用途 |
| --- | --- | --- |
| 维度字段 | `text` / `select` / `date` / `person` / `department` | 横轴、分类、筛选 |
| 指标字段 | `number` | 求和、排序、统计 |
| 说明字段 | `text` | 详情展示，不直接做 `valueKey` |

高风险提醒：

- 不要把长文本说明字段当指标字段
- 不要把纯文本金额字段直接拿去做图
- 不要等到报表阶段才发现字段类型不适合

## 3. 字段分层设计

### 3.1 主展示字段

主展示字段用于：

- 列表第一眼识别
- relation 关联值展示
- 关键词搜索主入口

常见建议：

| 业务对象 | 推荐字段 | 类型 | 设计建议 |
| --- | --- | --- | --- |
| 客户 | 客户名称 | `text` | 建议主展示、必填、可搜索 |
| 联系人 | 联系人姓名 | `text` | 建议主展示、必填、可搜索 |
| 商机 | 商机名称 | `text` | 建议主展示、必填、可搜索 |
| 工单 | 工单标题 | `text` | 建议主展示、必填、可搜索 |

### 3.2 状态字段

状态字段主要用于筛选、分组、流程推进。

| 常见字段 | 推荐类型 | 是否参与筛选 | 是否参与排序 | 说明 |
| --- | --- | --- | --- | --- |
| 客户状态 | `select` | 是 | 视情况 | 例如潜客、跟进中、成交、流失 |
| 商机阶段 | `select` | 是 | 视情况 | 例如初步接触、方案、报价、成交 |
| 跟进状态 | `select` | 是 | 否 | 更适合筛选，不一定适合排序 |
| 工单状态 | `select` | 是 | 否 | 通常用于列表过滤和看板分组 |

补充要求：

- `select` / `multiSelect` 字段不能只给类型，必须同时给出候选项列表
- 如果 AI 产出的字段设计里只有“客户状态：select”，没有“潜客 / 跟进中 / 成交 / 流失”这类选项值，说明设计还没完成

### 3.3 分类字段

分类字段用于统计分组和快速过滤。

| 常见字段 | 推荐类型 | 说明 |
| --- | --- | --- |
| 客户等级 | `select` | A/B/C 等级适合筛选 |
| 行业 | `select` 或 `text` | 稳定枚举用 `select`，不稳定类目可先用 `text` |
| 来源渠道 | `select` | 便于看不同渠道转化 |
| 标签 | 多选类字段 | 适合轻量标签分类 |

### 3.3.1 `select` / `multiSelect` 自定义选项模板

CLI 参数：

- `--type select` 或 `--type multiSelect`
- `--options <JSON 选项数组或逗号分隔选项列表>`

CLI 示例：

```bash
dimens-cli column create \
  --team-id TTFFEN \
  --project-id PUQUNFE \
  --sheet-id sh_customer \
  --label 提交状态 \
  --type select \
  --options '[{"label":"待提交","color":"bg-slate-100 text-slate-700"},{"label":"提交中","color":"bg-blue-100 text-blue-700"},{"label":"已提交","color":"bg-emerald-100 text-emerald-700"},{"label":"已驳回","color":"bg-rose-100 text-rose-700"}]'
```

```bash
dimens-cli column create \
  --team-id TTFFEN \
  --project-id PUQUNFE \
  --sheet-id sh_customer \
  --label 群组类型 \
  --type multiSelect \
  --options 系统群,业务群,临时群,项目群
```

API body 格式：

```json
{
  "label": "提交状态",
  "type": "select",
  "config": {
    "options": [
      { "label": "待提交", "color": "bg-slate-100 text-slate-700" },
      { "label": "提交中", "color": "bg-blue-100 text-blue-700" },
      { "label": "已提交", "color": "bg-emerald-100 text-emerald-700" },
      { "label": "已驳回", "color": "bg-rose-100 text-rose-700" }
    ],
    "dataSourceType": "manual",
    "dictionaryId": null
  }
}
```

说明：

- 当前先支持“自定义选项”模式，不扩展字典模式
- `--options` 推荐传 JSON 选项对象数组，CLI 会优先按对象数组映射为 `config.options`
- 旧的逗号分隔字符串仍兼容，但只适合最简单的文本选项
- 技能生成字段方案时，凡是 `select` / `multiSelect`，都应直接产出这组候选项，而不是留空
- 如果用户实际要的是“选人”或“选部门”，不要为了看起来像下拉就继续用 `select` / `multiSelect`

### 3.4 时间字段

时间字段决定列表能否做区间筛选和时间倒序。

| 常见字段 | 推荐类型 | 是否参与筛选 | 是否参与排序 | 说明 |
| --- | --- | --- | --- | --- |
| 创建时间 | `system` / `datetime` | 是 | 是 | 默认常用排序字段 |
| 最近跟进时间 | `datetime` | 是 | 是 | CRM 场景非常关键 |
| 预计成交日期 | `date` | 是 | 是 | 常用于商机到期提醒 |
| 下次回访时间 | `datetime` | 是 | 是 | 常用于待办筛选 |

### 3.5 数值字段

数值字段适合排序、统计和范围筛选。

| 常见字段 | 推荐类型 | 设计建议 |
| --- | --- | --- |
| 预计金额 | `number` | 建议支持排序 |
| 成交金额 | `number` | 建议支持排序和统计 |
| 评分 | `number` | 明确是否允许小数 |
| 优先级分值 | `number` | 用于排序，不建议用户手工乱填 |

补充说明：

- 这类字段如果后续要进入报表，优先作为 `valueKey`
- 不要把“100万”“高”“低”这类文本值伪装成数值指标

### 3.6 人员字段

人员字段通常用于负责人筛选和协作分工。

| 常见字段 | 推荐类型 | 设计建议 |
| --- | --- | --- |
| 销售负责人 | `person` | 建议支持筛选 |
| 跟进人 | `person` | 建议支持筛选 |
| 创建人 | `system` | 一般只读 |
| 更新人 | `system` | 一般只读 |

补充说明：

- 如果项目已经有自己的用户体系、内置角色、部门体系，且用户需求是“做一个人员下拉”，默认应理解为“配置人员字段”，而不是“创建静态下拉选项”
- 人员字段属于特殊情况，优先走真实用户选择语义，不要手工维护“张三、李四、王五”这类静态 `select` 选项

### 3.6.1 部门字段

部门字段用于部门归属、组织筛选和部门维度统计。

| 常见字段 | 推荐类型 | 设计建议 |
| --- | --- | --- |
| 所属部门 | `department` | 建议支持筛选 |
| 负责部门 | `department` | 建议支持筛选 |
| 发起部门 | `department` | 结合流程场景判断是否只读 |

补充说明：

- 部门字段也属于特殊情况，不要把真实组织结构字段退化成普通 `select`
- 如果字段值会随着组织架构调整而变化，或者需要和部门权限、部门筛选联动，更应该使用 `department`

### 3.7 备注字段

备注字段不是核心筛选字段，但常用于补充业务上下文。

| 常见字段 | 推荐类型 | 设计建议 |
| --- | --- | --- |
| 客户需求说明 | `text` | 适合详情展示 |
| 跟进备注 | `text` | 适合长文本记录 |
| 处理说明 | `text` | 不建议作为主筛选字段 |

## 4. relation 字段设计模式

### 4.1 一对一关系

例如联系人表里的“所属客户”。

| 设计项 | 建议值 |
| --- | --- |
| 字段名 | 所属客户 |
| 类型 | `relation` |
| 目标表 | 客户表 |
| 展示字段 | 客户名称 |
| 是否多选 | 否 |
| 是否双向 | 视业务决定 |
| 编辑视图字段 | 客户名称、客户编号、手机号 |

适用场景：

- 联系人归属一个客户
- 商机归属一个客户
- 工单归属一个项目

### 4.2 一对多关系

例如客户表里的“联系人列表”或商机表里的“关联跟进记录”。

| 设计项 | 建议值 |
| --- | --- |
| 字段名 | 联系人列表 |
| 类型 | `relation` |
| 目标表 | 联系人表 |
| 展示字段 | 联系人姓名 |
| 是否多选 | 是 |
| 是否双向 | 建议结合主从方向判断 |
| 编辑视图字段 | 联系人姓名、手机号、职位 |

适用场景：

- 一个客户有多个联系人
- 一个商机有多条跟进记录
- 一个项目挂多个售后工单

### 4.2.1 relation 字段模板

CLI 参数：

- `--type relation`
- `--target-sheet-id <目标表ID>`
- `--display-column-id <目标表展示字段ID>`
- `--bidirectional true|false`
- `--multiple true|false`

API body 格式：

```json
{
  "label": "字段名称",
  "type": "relation",
  "config": {
    "relationConfig": {
      "targetSheetId": "sh_目标表ID",
      "displayColumnId": "fld_显示字段ID",
      "bidirectional": false
    }
  }
}
```

说明：

- `--target-sheet-id` 对应服务端真实字段 `config.relationConfig.targetSheetId`
- `displayColumnId` 建议显式提供，否则关联值展示体验通常不稳定
- 行写入 relation 值前，仍然需要先确认目标表和展示字段

### 4.3 relation 字段的常见失败点

| 现象 | 常见原因 | 处理建议 |
| --- | --- | --- |
| 能关联但显示值看不懂 | 没指定正确展示字段 | 补 `displayColumnId` |
| 关联选择器难搜索 | `editViewFields` 太少 | 把名称、编号、手机号等一起纳入 |
| 一条记录只能选一个对象 | 多选能力没设计清楚 | 明确是否需要多选 relation |
| 主表能看到值，目标表不好追踪 | 是否双向没提前设计 | 视业务决定是否需要双向关系 |

## 5. 默认建模骨架

当用户说“帮我做一个系统”但没有指定业务域时，默认优先输出下面这套骨架，再根据业务对象替换字段名：

| 层级 | 默认内容 | 说明 |
| --- | --- | --- |
| 项目 | 1 个项目 | 系统容器 |
| 核心表 | 2-4 张主表 | 围绕核心业务对象 |
| 字段设计 | 每张表至少 6-12 个字段 | 要明确字段能力 |
| 关联关系 | 至少 1-3 条 relation | 保证多表联动 |
| 示例数据 | 每张表 3-5 行 | 便于立刻试用 |
| 查询案例 | 关键词 / 筛选 / 排序 / 视图 | 验证表是否真能用 |

## 7. 与其他 references 的关系

建议按下面顺序组合使用：

1. 先看本文件，明确字段怎样设计才可用。
2. 再看 `row-filters.md`，确认这些字段怎样参与搜索、筛选、排序。
3. 再看 `examples.md`，确认真实接口的请求路径、CLI 命令、出参结构。
4. 如涉及系统视图和项目级绑定，再看 `field-rules.md`。
