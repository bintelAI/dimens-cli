# dimens-manager Key 鉴权章节 接入边界说明

## 1. 文档目标

这份文档专门解决一个高频误区：

很多用户会把 API Key 能力理解成“系统已经有一套新的开放平台”。

实际上当前能力边界更准确的说法是：

1. 用 `apiKey + apiSecret` 换出现有用户 token
2. 再继续访问原有 `/app/*` 业务接口
3. 权限仍然继承绑定用户本身

因此它是登录方式扩展，不是新开放平台。

---

## 2. 三个必须先讲清的边界

### 2.1 不是新的权限体系

API Key 本身没有独立权限集。

换到 token 后：

- 能访问什么团队
- 能访问什么项目
- 能访问什么表、工作流、报表

仍然取决于绑定用户原本的权限。

### 2.2 不是新的资源模型

第三方系统换到 token 后，调用的仍然是现有业务接口，例如：

- `/app/org/*`
- `/app/mul/*`
- `/app/flow/*`
- `/app/report/*`

所以 Skill 在说明“第三方怎么接”时，不应该写成“去调用一套新的开放接口目录”。

### 2.3 不是绕过团队 / 项目隔离

即使 Key 登录成功，后续访问依然要继续命中：

- 团队成员关系
- 项目准入
- 资源权限

因此“登录成功但没权限”是合理现象，不是 Key 体系本身一定坏了。

---

## 3. 第三方接入时的标准解释

当用户说“我要给第三方系统接入”时，建议固定按下面顺序解释：

1. 先通过 API Key 登录换 token。
2. 再把 token 放进现有 `/app/*` 请求头。
3. 如果后续访问受限，继续按原有团队 / 项目 / 资源权限排查。

推荐表达模板：

```md
当前 API Key 能力不是新的开放平台，而是现有登录体系的扩展。

第三方系统先用 `apiKey + apiSecret` 换取现有用户 token，
再继续访问原有 `/app/*` 业务接口。

因此后续权限仍然继承该用户本身，而不是由 Key 单独决定。
```

---

## 4. 哪些问题优先查 Key，哪些问题优先查权限

### 4.1 优先查 Key 本身

以下问题优先查 Key：

- `api-key-login` 直接失败
- 同一组 Key 在某环境能用、另一环境不能用
- 提示过期、禁用、secret 不匹配
- 怀疑 IP 白名单限制

### 4.2 优先查用户权限

以下问题优先查用户权限：

- 登录成功，但 `project list` 看不到项目
- 登录成功，但 `sheet/row/flow/report` 接口无权限
- 不同 Key 登录结果一致地缺权限

因为这些现象通常说明：

- Key 认证成功了
- 但绑定用户本身就没有对应资源权限

## 5. 与其他 references 的关系

建议按下面顺序组合使用：

1. 先看 `login-flow.md`，确认登录换 token 链路是否正常。
2. 再看本文件，确认第三方接入和权限继承边界。
3. 最后看 `capability-status.md`，确认哪些能力已经 CLI 化，哪些仍然只是接口能力。
