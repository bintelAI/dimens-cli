# dimens-system-orchestrator 场景分类与执行细则

## 1. 结论

把 `dimens-system-orchestrator` 细分为场景是可行的，而且比只按“表格 / 权限 / 报表 / 画布”这种资源类型路由更稳定。

推荐固定为 5 类场景：

| 编号 | 场景 | 主要目标 | 输出形态 |
| --- | --- | --- | --- |
| 0 | 项目梳理场景 | 读取已有项目，梳理资源、结构、数据、权限和风险 | 项目盘点报告、资源清单、改造建议 |
| 1 | 新建项目场景 | 从 0 创建一个完整业务系统 | 项目、目录、表格、文档、画布、报表、示例数据、角色权限、工作流 |
| 2 | 修改项目内数据场景 | 在已有项目里更新表、字段、行、文档、报表、画布或权限 | 变更计划、读前快照、更新动作、回查结果 |
| 3 | 查询场景 | 查询已有项目资源、结构、数据、报表、权限或工作流状态 | 查询条件、命中结果、解释、下一步建议 |
| 4 | 分类路由场景 | 用户意图不完整或混合时，先归类再路由 | 场景判断、需要的上下文、下一跳章节 |

这 5 类不是互斥的执行域，而是总控层的入口判断。真实落地仍然进入 `dimens-manager` 对应章节，SDK 接入才进入 `dimens-sdk`。

## 2. 场景 0：项目梳理场景

### 2.1 适用输入

用户表达类似：

- “帮我梳理这个项目”
- “看看当前项目里有什么”
- “整理项目结构、表、文档、报表、权限”
- “根据这个项目生成说明文档”
- “检查这个项目还能怎么优化”

### 2.2 必须先拿到的上下文

| 信息 | 来源 |
| --- | --- |
| `teamId` | 用户链接、显式输入、当前认证上下文 |
| `projectId` | 用户链接、项目列表、显式输入 |
| 项目菜单树 | `sheet tree` |
| 项目详情 | `project info` |
| 表格清单 | 菜单树、`sheet info` |
| 文档清单 | 菜单树、`doc info` |
| 报表清单 | 菜单树、`report list` |
| 画布清单 | 菜单树、`canvas info` |
| 权限与角色 | `role`、`permission`、`row-policy`、`row-acl` |

如果用户只给了页面链接，先按链接规则解析：

```text
https://dimens.bintelai.com/#/TEAM_ID/PROJECT_ID/SHEET_ID?view=VIEW_ID
```

得到 `teamId / projectId / sheetId / viewId` 后再继续。文档页面链接里的 `sh_xxx` 默认先当菜单资源 `sheetId`，要用 `doc info --sheet-id sh_xxx` 换真实 `documentId`。

### 2.3 梳理维度

项目梳理不能只列表名，至少覆盖下面 8 层：

| 层级 | 要梳理什么 | 关注点 |
| --- | --- | --- |
| 项目容器 | 项目名称、类型、描述、封面、归属团队 | `teamId / projectId` 是否稳定 |
| 菜单目录 | folder、sheet、document、report、canvas 的层级 | 资源是否归位，是否有散落资源 |
| 表格模型 | 表、字段、视图、主展示字段、状态字段 | 是否可录入、可筛选、可排序 |
| 关联关系 | relation 字段、目标表、展示字段、双向关系 | 是否需要先取目标表 ID 和展示字段 ID |
| 示例数据 | 行数据数量、关键样例、数据完整性 | 是否足以验证查询和报表 |
| 文档资源 | 操作说明、制度、知识沉淀、版本链 | 是否误把文档建成表 |
| 报表资源 | 组件、数据源、参数、联动、查询结果 | 是否只是空壳 |
| 权限流程 | 角色、项目权限、表列行权限、工作流 | 是否存在越权或不可见风险 |

### 2.4 推荐输出结构

```md
## 项目概览
- 团队：TEAM_ID
- 项目：PROJECT_ID
- 项目定位：...

## 资源清单
| 类型 | 名称 | ID | 所属目录 | 状态 |

## 数据模型
| 表 | 主字段 | 状态字段 | 关联字段 | 视图 |

## 业务链路
- 核心对象：
- 关键状态：
- 主要流程：

## 权限与协作
- 角色：
- 数据可见范围：
- 高风险点：

## 问题与建议
| 优先级 | 问题 | 影响 | 建议 |
```

### 2.5 验收标准

- 能明确说清项目里有哪些资源。
- 能说清核心表之间的关系。
- 能指出文档、报表、画布是否已经落地。
- 能指出权限和协作风险。
- 不把“菜单资源 ID”和“业务内容 ID”混用。

## 3. 场景 1：新建项目场景

### 3.1 适用输入

用户表达类似：

- “帮我生成一个客户管理系统”
- “新建一个项目管理平台”
- “设计并创建一个审批系统”
- “从零搭一个售后工单系统”
- “创建菜单目录结构，包含表格、文档、画布、报表”

### 3.2 新建项目不是单个命令

新建项目场景必须按“系统交付物”理解，而不是只执行 `project create`。

默认交付物包括：

| 交付物 | 是否默认考虑 | 说明 |
| --- | --- | --- |
| 项目容器 | 是 | 系统根容器 |
| SVG 封面 / 图标 | 是 | 适合系统展示、模板化、品牌化入口 |
| 菜单目录 | 是 | folder 结构，避免资源散落 |
| 表格 | 是 | 结构化业务对象 |
| 在线文档 | 是 | 操作说明、制度、知识沉淀 |
| 业务场景画布 | 按需但强建议 | 有流程、多角色、状态流转时必须补 |
| 审批工作流画布 | 审批场景必须 | 只表达审批场景，不等同可执行流 |
| 报表 | 管理类系统默认考虑 | 经营分析、状态统计、趋势看板 |
| 示例数据 | 是 | 验证字段、关联、查询、报表 |
| 角色权限 | 涉及多人协作时必须 | 项目、表、列、行、资源权限 |
| 可执行工作流 | 审批或自动化时必须 | 真实定义、发布、挂载、运行验证 |

### 3.3 标准执行顺序

```text
1. 识别系统定位
2. 设计项目与菜单目录
3. 生成 SVG 封面并上传
4. 创建项目并写回封面 URL
5. 创建目录结构
6. 创建表格和默认视图
7. 创建字段和 relation
8. 录入基础表示例数据
9. 获取基础表行 ID / 字段 ID
10. 录入带关联的示例数据
11. 创建在线文档
12. 创建业务场景画布
13. 创建报表并跑预检链
14. 创建角色、权限和行级策略
15. 创建审批工作流或自动化工作流
16. 回查项目菜单、结构、数据、权限、报表、画布
```

### 3.4 系统定位设计

总控先输出项目方案，而不是直接跑命令：

| 维度 | 示例 |
| --- | --- |
| 系统名称 | 客户管理系统 |
| 主要角色 | 销售、销售主管、财务、管理员 |
| 核心对象 | 客户、联系人、商机、合同、跟进记录 |
| 生命周期 | 线索 -> 跟进中 -> 报价 -> 成交 -> 归档 |
| 核心动作 | 新建客户、分配负责人、记录跟进、推进商机、审批合同 |
| 关键报表 | 客户来源、商机漏斗、合同金额趋势 |
| 关键权限 | 销售看自己，主管看部门，管理员看全部 |

### 3.5 SVG 封面 / 图标

新建系统建议生成一张项目封面：

| 项 | 要求 |
| --- | --- |
| 尺寸 | `250x150px` |
| 文件类型 | `.svg` |
| MIME | `image/svg+xml` |
| 背景 | 淡色、低干扰 |
| 动效 | 轻量 `animate` / `animateTransform` |
| 内容 | 与系统主题相关，不要做纯装饰 |

执行链路：

```bash
dimens-cli upload file \
  --path ./project-cover.svg \
  --team-id TEAM_ID \
  --scene project-cover \
  --source material
```

上传后必须拿到 `url`，再写回项目封面或图标。只上传不写回不算完成。

### 3.6 创建菜单目录结构

菜单目录先于资源大规模创建。推荐至少按业务域分目录：

| 目录 | 放什么 |
| --- | --- |
| 业务数据 | 主业务表、从业务表 |
| 流程协作 | 审批表、任务表、处理记录表、画布 |
| 经营分析 | 报表、看板 |
| 项目文档 | 制度、说明、SOP、知识沉淀 |
| 系统配置 | 枚举表、基础档案表、权限说明 |

命令示例：

```bash
dimens-cli sheet create \
  --team-id TEAM_ID \
  --project-id PROJECT_ID \
  --name 业务数据 \
  --type folder
```

后续创建表格、文档、报表、画布时，能带 `--folder-id` 的必须显式带目录 ID。已有资源要归位时，使用：

```bash
dimens-cli sheet update RESOURCE_ID \
  --team-id TEAM_ID \
  --project-id PROJECT_ID \
  --folder-id FOLDER_ID
```

完成后必须 `sheet tree` 回查。

### 3.7 表格设计

表格设计先分主对象、从对象、配置对象：

| 类型 | 示例 | 建议 |
| --- | --- | --- |
| 主对象表 | 客户、项目、工单、申请单 | 至少有主展示字段、状态字段、负责人、时间字段 |
| 从对象表 | 联系人、跟进记录、处理记录、审批记录 | 通常通过 relation 指向主对象 |
| 配置表 | 分类、产品、服务项、费用项 | 稳定字典可用 select，不稳定或需维护可建配置表 |

每张表至少设计：

- 主展示字段：如客户名称、工单标题、申请单编号。
- 状态字段：必须给出候选项，不创建空 `select`。
- 人员字段：负责人、审批人、处理人优先用 `person`。
- 部门字段：所属部门、发起部门优先用 `department`。
- 时间字段：创建时间、截止时间、完成时间。
- 数值字段：金额、数量、评分，必须用 `number`，不要存成文本。
- 关联字段：必须明确目标表和展示字段。

### 3.8 关联表与示例数据顺序

涉及 relation 的示例数据，必须遵循“先基础表，后关联表”的顺序。

错误做法：

```text
直接给联系人表写入“所属客户=华东智造”
```

正确做法：

```text
1. 创建客户表
2. 创建客户名称字段
3. 写入客户示例行
4. 回查客户表行数据，拿到客户 rowId
5. 创建联系人表
6. 创建联系人所属客户 relation 字段，拿到 relation fieldId
7. 写入联系人示例行，relation 值使用目标客户 rowId 或接口要求的结构
8. 回查联系人表，确认关联值能展示
```

示例：

```bash
dimens-cli row page \
  --team-id TEAM_ID \
  --project-id PROJECT_ID \
  --sheet-id CUSTOMER_SHEET_ID \
  --page 1 \
  --size 20
```

拿到客户 `rowId` 后，再写联系人：

```bash
dimens-cli row create \
  --sheet-id CONTACT_SHEET_ID \
  --values '{"fld_name":"王明","fld_customer":["CUSTOMER_ROW_ID"]}'
```

如果后端 relation 值要求对象结构，要按当前字段 `relationConfig` 和行数据返回格式调整。不要只凭中文名称写关联。

### 3.9 示例数据要求

每张核心表建议至少 3 条示例数据，且要覆盖不同状态：

| 表 | 示例要求 |
| --- | --- |
| 客户表 | 不同等级、来源、负责人 |
| 商机表 | 不同阶段、金额、预计成交日期 |
| 工单表 | 不同优先级、状态、处理人 |
| 审批申请表 | 待提交、审批中、已通过、已拒绝 |
| 报表数据源表 | 至少能支撑分组、筛选、求和 |

批量写入优先使用：

```bash
dimens-cli row batch-create \
  --sheet-id SHEET_ID \
  --file ./data/rows.json \
  --batch-size 200
```

### 3.10 在线文档

新建系统默认至少补一份说明文档：

| 文档 | 作用 |
| --- | --- |
| 系统使用说明 | 面向普通用户 |
| 管理员配置说明 | 面向管理员 |
| 审批规则说明 | 面向审批系统 |
| 数据字典说明 | 面向实施和维护 |

在线文档走 `doc create / doc info / doc update`，不要误建成普通表。文档更新必须先读版本号，再带 `--version` 更新。

### 3.11 业务场景画布

出现下面任一条件必须补画布：

- 3 个以上角色协作。
- 核心对象有生命周期。
- 有审批、退回、撤回、超时、转交。
- 用户要“可视化业务流程”。

画布至少表达：

- 角色泳道。
- 业务对象。
- 状态节点。
- 条件分支。
- 异常路径。
- 数据回写。

保存型任务必须执行或说明：

```text
canvas create -> canvas info -> canvas save
```

### 3.12 报表

报表不应最后随手补一个空壳。创建前先确认：

| 项 | 示例 |
| --- | --- |
| 谁看 | 管理员、主管、个人 |
| 回答什么问题 | 当前待办、转化率、处理时长、金额趋势 |
| 数据源 | 哪些表和字段 |
| 维度 | 状态、部门、负责人、时间 |
| 指标 | 数量、金额、耗时、评分 |
| 参数 | 时间范围、部门、负责人 |

固定预检链：

```text
report create -> report preview -> report widget-add -> report query-widget -> report query
```

没有跑通 `report query`，不能说报表完成。

### 3.13 角色权限

多人协作系统默认要设计角色权限：

| 角色 | 典型权限 |
| --- | --- |
| 普通成员 | 进入项目、查看/编辑自己负责的数据 |
| 主管 | 查看本部门数据、分配任务、查看部门报表 |
| 财务 / 法务 | 只处理相关审批或合同字段 |
| 管理员 | 管理表结构、角色、全部数据 |
| 外部协作者 | 只看被授权资源或公开页面 |

落地顺序：

```text
role create -> permission create -> role assign-user -> row-policy create -> row-acl 按需补例外
```

不要只创建角色。角色没有绑定用户和权限记录时不会生效。

### 3.14 审批工作流

审批类系统要拆两层：

| 类型 | 作用 |
| --- | --- |
| 审批工作流画布 | 表达发起、分支、审批、拒绝、撤回、通知、回写 |
| 可执行审批工作流 | 真实创建、发布、项目挂载和运行 |

审批工作流至少设计：

- 触发表：申请单 / 合同 / 报销单。
- 触发字段：workflow 字段或状态字段。
- 入参：申请人、部门、金额、原因、附件。
- 条件：金额阈值、部门、风险等级。
- 审批人策略：直属上级、部门负责人、指定角色、指定人员。
- 动作：通过、拒绝、退回、转交、撤回。
- 回写：审批状态、审批摘要、审批时间、当前审批人。
- 通知：申请人、审批人、抄送人。

如果用户要求“能跑”，必须进入 `dimens-manager/references/workflow/references/approval-generation.md`。

### 3.15 新建项目完成判定

完成后至少回查：

| 回查项 | 命令 / 方式 |
| --- | --- |
| 项目详情 | `project info` |
| 封面写回 | `project info` 看 URL |
| 菜单归位 | `sheet tree` |
| 表结构 | `sheet info`、`column list` |
| 默认视图 | `view list` |
| 行数据 | `row page` |
| 关联数据 | 查询从表，确认 relation 展示 |
| 文档 | `doc info` |
| 报表 | `report query` |
| 画布 | `canvas info`、`canvas save` |
| 权限 | `role / permission / row-policy` 回查 |
| 工作流 | 工作流定义、发布、项目挂载、运行验证 |

## 4. 场景 2：修改项目内数据场景

### 4.1 适用输入

用户表达类似：

- “帮我修改这个表的数据”
- “把状态字段改成已完成”
- “给现有项目加一张表”
- “修改文档内容”
- “调整菜单目录”
- “给这个角色加权限”
- “更新报表组件”

### 4.2 更新安全链路

更新类任务必须遵循：

```text
先读取当前数据 -> 分析目标字段和版本 -> 形成最小变更 -> 提交更新 -> 回查验证
```

不能直接按用户口述猜字段 ID、行 ID、版本号。

### 4.3 修改类型与读取要求

| 修改类型 | 更新前必须读取 | 更新后必须回查 |
| --- | --- | --- |
| 项目信息 | `project info` | `project info` |
| 菜单目录 | `sheet tree` | `sheet tree` |
| 表字段 | `column list`、`sheet info` | `column list` |
| 表行数据 | `column list`、`row page` | `row page` |
| 单元格 | 目标行 `rowId`、`version`、目标 `fieldId` | `row page` |
| relation 单元格 | 目标表 ID、目标行 ID、relation fieldId | 从表查询展示 |
| 文档 | `doc info` 拿版本 | `doc info`、必要时 `doc versions` |
| 报表 | `report preview`、`query-widget` | `report query` |
| 画布 | `canvas info`、当前版本 | `canvas versions`、`canvas info` |
| 权限 | 角色、权限、策略现状 | 权限快照或策略列表 |

### 4.4 修改表内数据的细节

如果用户只给中文字段名，先通过 `column list` 映射到 `fieldId`。

如果用户只说“第一条”或“客户 A”，先用 `row page` 或筛选查询定位目标行，拿到：

- `rowId`
- 当前 `version`
- 当前字段值

然后再执行：

```bash
dimens-cli row set-cell \
  --sheet-id SHEET_ID \
  --row-id ROW_ID \
  --field-id FIELD_ID \
  --value 新值 \
  --version VERSION
```

如果是 relation 字段，不能把展示文本当真实值。必须先查询关联表拿到目标 `rowId`。

### 4.5 修改结构的边界

结构修改风险更高，默认只做最小必要变更：

- 新增字段可以执行。
- 修改字段类型要先评估历史数据兼容性。
- 删除字段不直接执行；如确需处理，线上业务数据删除必须另行确认。
- 批量修改行数据前先给出筛选条件和影响数量。

## 5. 场景 3：查询场景

### 5.1 适用输入

用户表达类似：

- “查询这个项目有哪些表”
- “看看客户表里有哪些数据”
- “查一下审批中的记录”
- “这个报表为什么没数据”
- “这个人为什么看不到表”
- “列出所有角色权限”

### 5.2 查询分类

| 查询类型 | 入口 | 说明 |
| --- | --- | --- |
| 项目查询 | `project info/list` | 查项目容器 |
| 菜单查询 | `sheet tree` | 查目录和资源归位 |
| 表结构查询 | `sheet info`、`column list`、`view list` | 查字段和视图 |
| 行数据查询 | `row page` | 查业务数据 |
| 关联查询 | 先查字段，再查主从表数据 | 不凭展示名推断 |
| 文档查询 | `doc info`、`doc versions` | 文档链接先换真实 `documentId` |
| 报表查询 | `report preview/query-widget/query` | 判断空壳、映射错误、数据为空 |
| 画布查询 | `canvas info/versions` | 查版本和保存内容 |
| 权限查询 | `role/permission/row-policy/row-acl` | 区分项目、表、列、行、协同 |
| 工作流查询 | workflow 章节 | 查定义、挂载、运行 |

### 5.3 查询输出要求

查询结果必须回答 4 件事：

1. 查询条件是什么。
2. 查到了什么。
3. 这些结果说明什么。
4. 下一步如果要修改，应该进入哪个场景和章节。

例如：

```md
我查的是 `客户表` 的公开默认视图和前 20 行数据。
结果显示有 5 个字段、3 条数据，其中 `所属客户` 是 relation 字段。
如果后续要新增联系人示例数据，需要先取客户表目标行 ID，再写联系人表的 relation 值。
```

### 5.4 查询场景的风险

- 不要把空结果直接解释为“没有权限”，也可能是筛选条件错误或数据源为空。
- 不要把能查到行数据解释为协同权限也正确，协同广播有独立风险。
- 不要只返回原始 JSON，要给业务解释。
- 查询结果用于后续修改时，要保留 ID、版本、字段映射关系。

## 6. 场景 4：分类路由场景

### 6.1 适用输入

用户表达不完整、混合或不确定时，先分类：

- “帮我处理一下这个项目”
- “这个系统有点乱”
- “帮我做一下权限和报表”
- “我想搞一个业务流程”
- “看看能不能自动化”

### 6.2 分类判断表

| 用户关键词 | 优先归类 | 下一跳 |
| --- | --- | --- |
| 梳理、盘点、整理、看看有什么 | 场景 0 | 项目梳理 |
| 新建、生成、创建、搭一个系统 | 场景 1 | 新建项目 |
| 修改、更新、调整、补充、移动 | 场景 2 | 修改项目内数据 |
| 查询、查看、列出、为什么没数据 | 场景 3 | 查询 |
| 不确定、混合、多目标 | 场景 4 | 先拆意图再路由 |

### 6.3 场景冲突时的优先级

| 情况 | 处理 |
| --- | --- |
| 用户给已有链接并说“修改” | 优先场景 2，先读取现状再改 |
| 用户说“生成系统”但已给 `projectId` | 先判断是场景 1 的“在已有项目初始化”，不是新建项目容器 |
| 用户说“查询并修改” | 先场景 3 查询，再进入场景 2 修改 |
| 用户说“新建系统并配置权限” | 场景 1，但权限前置设计 |
| 用户说“审批流程” | 场景 1 或 2，并拆画布与可执行工作流 |
| 用户说“API / SDK / 前端接入” | 跳 `dimens-sdk`，不要留在总控里写代码 |

### 6.4 最小澄清问题

当缺少关键上下文时，最多问 3 个问题：

1. 是要梳理已有项目，还是从零新建项目？
2. 是否已有 `teamId / projectId` 或维表链接？
3. 这次只要方案，还是要我直接执行创建 / 修改？

如果能从链接、命令上下文、用户描述推断，就不要反复追问。

## 9. 高风险细节清单

- 关联示例数据必须先拿目标表 `rowId`，不要直接写展示文本。
- relation 字段必须先确认目标表 `sheetId` 和展示字段 `fieldId`。
- 创建目录不会自动归位已有资源；新资源带 `--folder-id`，旧资源用 `sheet update --folder-id`。
- 在线文档的 URL 资源 ID 常是 `sheetId`，不是 `documentId`。
- 文档更新必须先读版本。
- 报表创建后不跑 `report query` 不能算完成。
- 画布保存必须有版本内容，不能只创建空壳。
- 审批工作流画布不等于可执行审批工作流。
- 角色创建不等于授权完成，必须绑定用户和权限记录。
- 行级权限和协同广播不能只验证 HTTP 行查询。
- 人员和部门字段不要退化成普通 select。
- 金额、数量、评分等报表指标必须用 number。
- 修改数据前必须拿 `rowId / fieldId / version`。
