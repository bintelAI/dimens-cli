# 新建项目维表设计文档模板

本文档用于约束 `dimens-system-orchestrator` 在处理“创建新项目 / 新建系统 / 根据一句话或文档搭建项目”类需求时的前置设计输出。

核心规则：

- 不管用户只给一句话，还是给完整需求文档，都必须先归一化成本文档格式。
- 设计文档确认前，不直接进入 `dimens-cli` 创建项目、表格、字段、报表、画布或权限配置。
- 本文档面向维表智联落地，不是通用 PRD；每一节都要能映射到项目、目录、表、字段、视图、文档、报表、画布、权限、工作流和验收命令。
- 如果用户信息不足，先给“基于当前信息的最小可执行设计草案”，并明确待确认项，不要停留在抽象提问。
- 输出时必须保持资源数量、菜单树、资源清单、表格设计和验收标准一致；不要出现“前面 6 张表，后面 5 张表”这类口径冲突。
- 所有 `select / multiSelect` 候选项必须使用前端可识别的颜色 class，例如 `bg-blue-100 text-blue-700`；不要输出 `blue / green / red` 这类颜色简写。
- 维表智联已经有自有用户体系、角色体系、权限体系和部门体系；凡是负责人、参与人、审批人、所属部门、负责部门、角色授权、行级权限等场景，优先复用内置体系，不要额外创建“用户表 / 角色表 / 部门表”或把人员部门做成普通下拉；部门字段当前禁止生成 `department` 类型，统一使用 `text` 保存部门名称。
- 分类、状态、优先级、来源、阶段、标签等字段要结合多维表格特性判断：选项少、稳定、只用于筛选/统计时优先用 `select / multiSelect`；只有需要层级维护、独立权限、扩展属性、频繁增删改、大量选项或被多表复杂复用时，才设计独立分类表。
- 目录名称、菜单名称、表名、文档名、画布名和报表名必须从用户业务语义中生成；模板里的“基础数据 / 业务处理 / 流程画布 / 报表看板 / 操作说明”等只是假设示例，禁止照抄为真实输出。
- 权限设计必须覆盖菜单/资源权限、表格权限、字段级读写权限和行级权限；每个菜单资源都要说明哪些角色可见、可编辑，每张表都要说明字段可见性和只读字段，不要只写“管理员/成员”这种粗粒度角色描述。
- 报表设计必须有数据源和样例数据支撑；设计阶段就要说明哪些字段提供维度、指标、筛选参数，执行阶段 `preview / query-widget / query` 为空时必须回到数据、参数或字段映射修正。
- 建表后如果平台自动生成默认“名称”字段，必须在设计和执行中说明处理方式：改名复用为主展示字段，或确认无依赖后清理；不要留下无业务意义的默认字段。
- 角色设计只创建业务角色，不重复创建平台内置超级管理员、系统管理员、管理员、编辑者、查看者、公开角色；业务角色创建后必须配置权限并绑定用户。
- 执行计划必须包含“一次性生成验收闸门”：目录归位、逐表真实 `fieldId`、写后 `row page` 验证、报表预检和最终全量回查。只要存在空目录、空表、行 `data:{}` 或空报表，就不能输出“已完成”。
- 建表计划必须先拆依赖层级：无依赖表可有限并发创建，被引用基础表先创建并写入样例数据，有单向绑定 / relation 依赖的表最后创建。示例数据必须对照表清单逐表写入，不能遗漏任何一张需要案例数据的表。

## 0. 已有文档结构校验

当用户在新建项目场景中提供已有需求文档、PRD、Markdown、Word 摘要、截图文字或链接内容时，先做结构校验，再决定是否能执行。不要因为文档篇幅很长就默认合格，核心是它能否直接指导维表智联落地。

### 0.1 合格判定

已有文档至少要能回答下面问题，才算基本符合本模板：

| 检查项 | 必须包含的内容 | 缺失时的处理 |
| --- | --- | --- |
| 系统定位 | 系统名称、目标、使用者、管理对象、业务结果 | 先补系统定位 |
| 角色协作 | 业务角色、职责、数据可见范围、可编辑范围 | 先补角色与协作边界 |
| 菜单树 | 按业务命名的目录、表、文档、画布、报表归属 | 先重写菜单树，禁止照抄模板示例 |
| 资源清单 | 项目、表格、文档、报表、画布、工作流、权限交付物 | 先补资源清单 |
| 核心对象 | 主对象、从对象、生命周期、状态流转 | 先补对象模型 |
| 表字段 | 每张表的字段名、字段类型、选项、人员/部门/relation 字段 | 先补字段设计 |
| relation | 关联目标表、展示字段、创建顺序、示例数据依赖 | 先补关联设计 |
| 选项/分类 | 哪些用 `select / multiSelect`，哪些才建分类表 | 先做多维表格建模判断 |
| 示例数据 | 每张核心表样例，报表数据源样例，关联行 ID 获取顺序 | 先补样例数据计划 |
| 报表口径 | 数据源、维度、指标、筛选参数、空数据排查 | 先补报表设计 |
| 画布/文档 | 文档内容范围，业务画布/审批画布表达内容 | 先补非表格交付物 |
| 权限矩阵 | 菜单资源权限、表权限、字段读写权限、行级权限 | 先补权限设计 |
| 执行计划 | 认证、创建、回查、验收顺序 | 先补执行计划 |
| 验收标准 | 资源 ID、菜单归位、字段回查、数据查询、报表/画布/权限回查 | 先补验收标准 |

### 0.2 不合格处理

如果已有文档缺少上表任一关键项，先输出或写入一份新的维表设计文档。重写时保留用户原始业务语义，但要转换成本文档结构，补齐维表智联需要的项目、菜单、资源、字段、权限和验收口径。

重写原则：

- 原文有价值的业务规则要保留，不要为了套模板丢掉业务约束。
- 原文是通用 PRD 时，要转成维表资源设计，不要照搬“功能模块、页面、接口”结构。
- 原文没有给出的内容可以用“默认假设 / 待确认项”标注，但不能省略表格、报表、权限和验收章节。
- 只有当重写后的文档能支撑执行顺序和 CLI 回查，才能继续创建项目。

### 维表内置体系复用

维表智联自带用户、角色、权限和部门体系。设计时先判断能否复用内置能力，不要把平台已有能力重复建成业务表。

| 业务需求 | 优先使用 | 不建议做法 | 说明 |
| --- | --- | --- | --- |
| 负责人、处理人、参与人、审批人 | `person` 字段 | 建“人员表”或普通 `select` 人员下拉 | 人员来自项目/团队用户体系，便于权限、筛选、协作联动 |
| 所属部门、负责部门、发起部门 | `text` 字段保存部门名称 | 建“部门表”、普通 `select` 部门下拉或 `department` 字段 | 当前 Web 前端对 `department` 类型支持不完整，先用部门名称文本承载筛选和报表维度 |
| 管理员、主管、成员、访客 | 角色体系 + 权限配置 | 建“角色字典表” | 真实访问控制应落到 role / permission / row-policy |
| 只能看自己 / 本部门数据 | 行级策略 | 在业务表里手写权限逻辑说明 | 权限必须能转成 row-policy 条件 |

只有当用户明确需要维护外部人员、外部客户组织、供应商组织、非维表账号联系人时，才考虑独立业务表。
```

### 6.2 菜单目录设计

目录和菜单名称必须根据当前业务系统来命名，不要照抄模板示例。命名优先使用用户业务语言，例如“客户池、商机跟进、合同回款、售后工单、设备台账、巡检计划、费用报销、项目交付”等。只有业务没有明显分组时，才使用简洁的通用目录名。

下面表格是结构模板，不是目录命名模板：

| 目录 | 资源类型 | 资源名称 | 创建方式 | 备注 |
| --- | --- | --- | --- | --- |
| [业务目录名称] | folder | [目录资源名称] | 新建 / 复用 | 说明该目录承载哪些业务资源 |
| [业务目录名称] | folder | [目录资源名称] | 新建 / 复用 | 说明该目录承载哪些业务资源 |
| [业务目录名称] | folder | [目录资源名称] | 新建 / 复用 | 说明该目录承载哪些业务资源 |

命名示例，仅用于理解，不可直接照抄：

| 系统类型 | 更合适的业务目录 | 不推荐的机械目录 |
| --- | --- | --- |
| CRM | 客户池、联系人、商机推进、合同回款、销售看板 | 基础数据、业务处理 |
| 工单系统 | 工单受理、处理跟踪、客户反馈、服务统计、规则说明 | 基础数据、业务处理 |
| 项目管理 | 项目立项、任务协作、里程碑、风险问题、项目报表 | 基础数据、业务处理 |
| 费用报销 | 报销申请、审批流转、付款记录、费用分析、制度说明 | 基础数据、业务处理 |

### 6.3 菜单树结构

用树形结构表达资源归属。目录、表、文档、画布、报表都要出现在树里；如果某类资源不需要创建，写“无”或从树里移除，不要保留空壳。
树里的目录和菜单必须是本业务系统的真实命名，不要使用模板占位名称；如果生成后看不出业务领域，说明菜单树不合格。

```text
[项目名称]
├── [业务目录 A，例如客户池 / 工单受理 / 项目立项]
│   ├── [该业务目录下的表格或文档]
│   └── [该业务目录下的表格或文档]
├── [业务目录 B，例如商机推进 / 处理跟踪 / 任务协作]
│   ├── [主业务表]
│   └── [从表 / 明细表 / 记录表]
├── [业务目录 C，例如合同回款 / 客户反馈 / 风险问题]
│   └── [相关资源]
├── [业务分析目录，例如销售看板 / 服务统计 / 项目报表]
│   └── [报表资源]
└── [业务说明目录，例如销售SOP / 服务规则 / 项目制度]
    └── [在线文档资源]
```

### 6.4 资源清单

资源清单必须和菜单树、后续表格/文档/画布/报表章节保持一致。生成设计文档后，要回头检查数量是否一致。

| 资源类型 | 资源名称 | 所属目录 | 是否创建 | 依赖资源 | 验收方式 |
| --- | --- | --- | --- | --- | --- |
| 表格 | [业务表名] | [业务目录名称] | 是 |  | `sheet info / column list / view list` |
| 在线文档 | [业务文档名] | [业务目录名称] | 是 / 按需 |  | `doc info` |
| 业务场景画布 | [业务画布名] | [业务目录名称] | 按需 | 表格字段、状态模型 | `canvas validate / canvas info / canvas versions` |
| 报表 | [业务报表名] | [业务目录名称] | 按需 | 数据表、示例数据 | `report preview / query-widget / query` |

### 6.5 是否需要独立分类表

不要看到“分类 / 类型 / 来源 / 优先级 / 状态 / 标签”就默认建表。先判断它适合字段选项，还是适合独立资源。

| 判断问题 | 适合 `select / multiSelect` | 适合独立分类表 |
| --- | --- | --- |
| 选项数量 | 少量稳定选项，通常 20 个以内 | 选项很多，后续持续扩充 |
| 维护频率 | 很少改，由管理员偶尔调整 | 需要业务人员频繁增删改 |
| 层级结构 | 无层级 | 需要父子分类、多级分类 |
| 扩展属性 | 只有名称和颜色 | 需要编码、负责人、SLA、说明、排序、启停、权限等属性 |
| 复用范围 | 主要在一张表内使用 | 被多张表 relation 复用且需要统一维护 |
| 权限要求 | 不需要单独控制谁能维护选项 | 需要独立权限、审计或启停控制 |
| 报表口径 | 只做分组统计 | 需要分类自身维度、关联属性或历史沉淀 |

输出设计文档时，涉及分类字段必须先给出结论：

| 分类项 | 推荐方案 | 理由 | 候选项 / 独立表字段 |
| --- | --- | --- | --- |
| 状态 | select | 生命周期状态少且稳定 | 待处理、处理中、已完成 |
| 标签 | multiSelect | 一条记录可有多个标签 | 重要、风险、需跟进 |
| 产品分类 | 独立分类表 | 需要层级、编码、负责人、启停 | 分类名称、父级分类、编码、状态 |
```

要求：

- 新建资源必须规划目标目录，避免资源散落。
- 如果目录无法提前确认，标记为“待归位”，执行后必须 `sheet tree` 回查。
- 菜单目录设计、菜单树结构、资源清单和验收标准里的资源名称及数量必须一致。
- 创建表、文档、画布、报表时优先带目标目录；但 `sheet create --folder-id` 不能作为最终归位证据，执行计划中必须包含创建后 `sheet move --folder-id` 补偿检查和 `sheet tree` 回查。
- 如果输出中出现“基础数据 / 业务处理 / 流程画布 / 报表看板 / 操作说明”这类模板化目录，必须先自检：它是否真的是用户业务语言；如果不是，立即改成业务名称。

字段类型注意：

- 稳定枚举用 `select` / `multiSelect`，并给出对象数组形式的 `options`。
- `options` 每项必须包含唯一 `id`、非空 `label`、合法 `color`；颜色使用 `bg-blue-100 text-blue-700` 这类前端色池 class，或 `custom:{"bg":"#xxxxxx","text":"#xxxxxx"}`。
- 人员用 `person`，部门当前用 `text` 保存部门名称；不要生成 `department` 类型。
- 角色、权限、项目准入、资源权限和行级可见性使用维表内置 `role / permission / row-policy` 体系，不要设计成普通业务字段替代真实权限。
- 状态、阶段、优先级、来源、简单分类等少量稳定选项，优先用 `select`；一条记录允许多个标签或分类时，优先用 `multiSelect`。
- 只有分类需要层级、启停、编码、负责人、SLA、说明、排序、权限、审计，或被多张表复杂复用时，才创建独立分类表并通过 `relation` 引用。
- 表间引用用 `relation`，不要退化成文本字段。
- 金额、数量、评分等用数值类型，日期用日期类型。
- 状态字段必须服务于视图、报表或流程，不要只作为展示文字。
- 系统字段、自动编号、创建人、创建时间、更新时间等字段要明确“系统内置 / 需 CLI 支持确认 / 本期改用普通字段”，不要在执行时盲写。
- 报表需要计算日期差、完成率、超时状态时，先判断产品是否支持公式或聚合；不确定时补冗余数值字段，例如“处理时长(小时)”或“是否超时”。

### 8.2 选项字段与分类表决策

每个“分类 / 类型 / 来源 / 优先级 / 标签 / 阶段 / 状态”字段都要先做决策，不要直接建分类表。

```md
### 选项字段与分类表决策

| 字段 / 分类项 | 业务含义 | 推荐类型 | 是否独立成表 | 决策理由 | 候选项或表字段 |
| --- | --- | --- | --- | --- | --- |
| 状态 | 生命周期状态 | select | 否 | 状态少且稳定，主要用于筛选、视图、报表 | 待处理、处理中、已完成 |
| 优先级 | 处理优先程度 | select | 否 | 少量稳定枚举 | 低、中、高、紧急 |
| 标签 | 多标签标记 | multiSelect | 否 | 一条记录可多个标签，但不需要独立属性 | 重点、风险、需复盘 |
| 分类 | 业务分类 | select / relation | 待判断 | 简单分类用 select；需要层级/编码/启停/负责人时建表 |  |
```

判断原则：

- `select`：单值、少量、稳定、只需要名称和颜色。
- `multiSelect`：多值、少量、稳定、只需要名称和颜色。
- `relation + 分类表`：分类本身需要字段、权限、层级、启停、审计、排序、负责人、SLA 或跨表复杂复用。
- `text`：开放输入、不可预知、不会作为固定筛选项。
- 不要为了“看起来专业”把简单状态、来源、优先级都拆成表；这会增加菜单复杂度、relation 写入成本和用户维护成本。

字段权限值建议使用：`不可见`、`可见 / 只读`、`可见 / 可编辑`。如果字段受流程控制，写明“由工作流写回 / 人工不可改”。

### 表级权限落地设计

| 角色 | 目标表 | dataAccess | canRead | canWrite | 字段可见性 | 字段只读 | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 管理员 | [表名] | full_rw | true | true | 全部可见 | 系统字段只读 | 可配置字段和视图 |
| 业务负责人 | [表名] | private_rw / scoped_rw | true | true | 敏感字段按需隐藏 | 系统字段、审批摘要只读 | 叠加部门行级策略 |
| 执行人员 | [表名] | private_rw | true | true | 隐藏敏感字段 | 状态、负责人、系统字段只读 | 叠加“负责人=当前用户” |
| 外部协作者 | [表名] | limited_submit / readonly | 按需 | 按需 | 仅开放提交字段 | 大多数字段只读 | 高风险，执行前确认 |

### 高风险权限点
- 外部访问：
- 公开分享：
- 跨部门可见：
- 敏感字段：
- 行级策略：

### 行级策略落地设计
需要“只能看自己 / 只能看本部门 / 只能改自己负责的数据”时，必须补下面表格。不要只写中文规则，必须能转成 `row-policy create` 的条件。

| 角色 | 目标表 | 动作 | 条件字段 | 条件表达式 | 优先级 | 是否启用 | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 执行人员 | 主业务表 | view / update | 负责人字段 fieldId 待执行时回填 | `负责人 contains {{currentUser}}` | 10 | 是 | 执行前先确认 person 字段真实结构 |
| 业务负责人 | 主业务表 | view / update | 部门字段 fieldId 待执行时回填 | `所属部门 contains {{currentUser.department}}` | 20 | 是 | 执行前确认部门上下文 |

### 权限验收
- `role create` 后必须 `role assign-user`，否则角色不会自动生效。
- 权限验收不能只看 `role list`；还要回查 `permission list / row-policy list / myPermissions`。
- 菜单/资源权限必须回查非表资源的可见和可编辑配置，文档、画布、报表、自定义页面不能只依赖项目级权限。
- 表格字段权限必须回查 `columnVisibility / columnReadonly`，确认不同角色下字段可见性和只读状态符合设计。
- 如果涉及协同编辑，还要说明前端权限快照或协同权限快照需要刷新验证。

### 查询用例
| 用例 | 查询方式 | 预期结果 |

| 关键词搜索 | 按名称 / 编号搜索 | 命中目标记录 |

| 报表预检 | 跑 `report preview / query-widget / query` | 每个组件都有数据或明确为空的业务原因 |

### 示例数据要求
- 每张核心表至少 3 条案例数据。
- 有 relation 的表，必须先创建基础表数据，再创建关联数据。
- 有单向绑定的表必须在依赖目标表、目标字段和必要目标行都回查完成后再创建和写入数据。
- 生成示例数据前必须列出完整表清单，并为每张需要样例数据的表标注 JSON 文件路径、预计行数和验收命令；不允许遗漏任何一张表。
- 报表数据要能覆盖至少 2 个状态、2 个负责人或 2 个时间段。

### 一致性检查
| 检查项 | 结果 | 修正动作 |
| --- | --- | --- |
| 场景归类里的表数量 = 表格设计数量 = 资源清单数量 = 验收标准数量 | 是 / 否 | 不一致时先统一数量 |
| 菜单树里的资源都在资源清单中出现 | 是 / 否 | 补资源或从树里移除 |
| 每个 relation 的目标表都先于源表创建 | 是 / 否 | 调整创建顺序 |
| 每个报表指标都有可查询字段支撑 | 是 / 否 | 补字段或降低指标范围 |
| 报表数据源表有支撑样例数据 | 是 / 否 | 补覆盖维度、指标、筛选的样例数据 |
| 权限中文规则能转成 row-policy 条件 | 是 / 否 | 补字段 ID 和条件表达式 |
| 只创建业务角色且跳过内置角色 | 是 / 否 | 删除重复内置角色设计，改用平台内置角色 |
| 每个菜单资源都有读写权限设计 | 是 / 否 | 补菜单 / 资源权限矩阵 |
| 每张表都有字段级读写权限设计 | 是 / 否 | 补字段权限矩阵和 columnVisibility / columnReadonly |
```

## 19. 使用规则

1. 新建项目场景必须先生成或补全本文档，再进入真实执行。
2. 如果用户给的是长文档，先提炼到本文档结构，不要直接照文档逐段创建资源。
3. 如果用户说“直接创建”，也要先输出最小可执行设计草案；只有用户明确授权继续执行后，才进入创建命令。
4. 设计文档里的表、字段、视图、权限、报表和画布必须能对应到后续 `dimens-manager` 章节。
5. 任何字段设计都要优先保证前端可渲染、CLI 可创建、后续可查询。
6. 对于不确定的信息，写默认假设和执行前确认点，不要编造真实 ID、人员、部门或外部接口。
7. 生成文档后必须做一次自检：资源数量一致、菜单树完整、字段颜色合法、relation 顺序正确、权限可转成策略、报表指标有字段支撑、画布验收链完整。
8. 目录和菜单命名必须业务化；不要把模板里的示例目录名、示例资源名、示例菜单树当作最终答案。
9. 权限设计必须拆到四层：菜单/资源权限、表格权限、字段读写权限、行级权限；缺少菜单权限表或字段权限表时，设计文档不完整。
