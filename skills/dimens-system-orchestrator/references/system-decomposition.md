# dimens-system-orchestrator 系统拆解指南

## 1. 拆解目标

这份文档用于把“生成一个 XX 系统”拆成可执行的模块，而不是直接跳到单个表或单个接口实现。

系统总控只输出系统级拆解、执行顺序、路由和验收证据，不在这里展开具体字段命令、接口代码或 SDK 示例。项目内落地细节进入 `dimens-manager`，接入代码进入 `dimens-sdk`。

## 2. 默认拆解主线

收到系统级需求时，默认先拆下面主线维度：

1. 系统定位
2. 团队与项目上下文
3. 系统交付物边界
4. 核心业务对象
5. 项目与表结构
6. 项目内文档 / 说明页
7. 字段与关联关系
8. 示例数据与查询案例
9. 业务场景画布

这些是默认主线，不应该一上来就被权限、流程、报表抢走重点。

## 3. 可选扩展维度

只有当用户继续要求时，再向下扩展：

1. 角色与权限
2. 流程与自动化
3. 审批工作流画布和可执行审批流
4. 报表与看板
5. 外部接入

## 4. 每个维度到底要拆什么

### 4.1 系统定位

必须先判断：

- 这是一个什么系统
- 主要服务哪些角色
- 当前重点是先搭数据系统，还是先搭流程系统

默认输出至少要有：

- 系统名称
- 核心目标
- 主要使用者

### 4.2 团队与项目上下文

必须先判断：

- 这是单团队单项目，还是多团队多项目场景
- 用户是团队管理员、项目管理员，还是普通成员/访客
- 系统是作为团队内共享系统，还是特定项目下的业务系统
- 需不需要部门、负责人、项目成员的层级管理

默认要带着下面两个问题继续往下拆：

1. 哪些资源属于 `teamId` 级别隔离
2. 哪些资源属于 `projectId` 级别隔离

### 4.3 核心业务对象

必须拆清：

- 主对象
- 从对象
- 对象之间的关联关系
- 对象的生命周期状态
- 对象对应的核心操作

系统级 Skill 不需要一开始就把字段穷举到最细，但至少要输出：

| 维度 | 应输出内容 |
| --- | --- |
| 主对象 | 系统最核心的数据实体 |
| 从对象 | 围绕主对象服务的附属实体 |
| 关系 | 一对多、多对多、阶段流转、归属关系 |
| 状态 | 新建、处理中、完成、关闭等状态模型 |
| 动作 | 创建、分配、审批、回访、归档等核心动作 |

### 4.3.1 系统交付物边界

完整系统默认至少判断下列交付物，不要把“系统搭建”收缩成单表创建：

| 交付物 | 默认判断 | 落地章节 |
| --- | --- | --- |
| 项目容器 | 必须 | `dimens-manager/references/project/overview.md` |
| 菜单目录 | 必须 | `dimens-manager/references/project/overview.md` |
| 表格与视图 | 必须 | `dimens-manager/references/table/overview.md` |
| 在线文档 | 默认考虑 | `dimens-manager/references/project/overview.md` |
| 报表看板 | 管理类系统默认考虑 | `dimens-manager/references/report/overview.md` |
| 业务场景画布 | 多角色、流程、审批类系统必须 | `references/business-canvas-flow.md` |
| 权限 | 多人协作、部门隔离、公开访问时必须 | `dimens-manager/references/permission/overview.md` |
| 可执行工作流 | 审批、自动化或 AI 节点运行时按需 | `dimens-manager/references/workflow/overview.md` |
| SDK / 外部接入 | 开发者集成时才进入 | `dimens-sdk` |

### 4.4 项目与表结构

默认至少要拆到：

- 项目名称
- 核心表
- 辅助表
- 是否需要文档或说明页

总控 Skill 到这一步，应该优先把系统收口成“项目 -> 表”而不是直接掉进权限或流程细节。

### 4.4.1 项目内文档 / 说明页

必须判断：

- 这个系统是否需要在线文档、产品说明页、知识沉淀页、操作手册页
- 这类资源是普通静态说明，还是需要多人协作编辑
- 如果是在线文档，应优先走文档主链，而不是误建成多维表格

补充原则：

- 项目资源不要只考虑“表格 + 报表”，文档同样是核心资源
- 当前在线文档编辑器采用 TipTap 富文本链路，文档主链应优先使用 `doc create / doc info / doc update / doc delete`
- 文档型项目或带说明页的业务系统，项目初始化时至少应补一份在线文档
- 文档资源不要被当成静态一次性产物；制度、说明、知识沉淀这类内容通常还要持续更新
- 如果系统里的文档承担长期演进内容，还要纳入版本链：`doc versions / doc version / doc restore`

### 4.5 字段与关联关系

默认至少要拆到：

- 主展示字段
- 状态字段
- 时间字段
- 关联字段
- 哪些字段要参与搜索、筛选、排序

### 4.6 示例数据与查询案例

默认至少要拆到：

- 每张表至少 3 条案例数据
- 至少 3 类查询案例：关键词搜索、字段筛选、排序 / 关联查询

如果这里不先拆清，后面的表结构往往只是“看起来像业务模型”，但无法直接使用。

### 4.7 角色与权限

必须至少拆到下面 4 层：

1. 谁能进入团队/项目上下文
2. 谁能看到系统模块
3. 谁能看或改具体数据
4. 谁能在协同和广播链路里继续看到数据变化

如果系统里有“公开访问”、“客户查看”、“外部协作者”等角色，必须额外标记为高风险项，因为这些角色经常会踩到项目准入、表级权限、列级权限、行级权限和协同广播的叠加边界。

### 4.8 流程与自动化

必须拆出：

- 哪些动作需要审批
- 哪些动作可以自动化
- 哪些动作只是提醒，不应该设计成强审批
- 哪些动作适合接 AI 分析或文本生成

这里最容易犯的错误是把所有流程都丢给工作流。总控 Skill 应该先判断这个流程到底属于：

- 审批流
- 自动化流
- AI 分析流
- 纯数据录入流程

如果流程涉及多角色协作、明显状态流转或审批节点，必须先补业务场景画布。画布用于表达“谁在什么阶段做什么、数据如何流转、异常如何处理”，再决定是否需要真实工作流。

### 4.8.1 业务场景画布

业务场景画布必须拆出：

- 角色泳道：发起人、处理人、审批人、财务、管理员、外部客户等
- 业务对象：表单、工单、合同、采购单、报销单、商机等
- 状态流转：草稿、待处理、审批中、已完成、已拒绝、已撤回
- 条件分支：金额、风险、优先级、部门、合同类型等
- 异常路径：退回、拒绝、撤回、超时、转交、人工兜底
- 数据沉淀：回写字段、审批摘要、报表统计口径

系统级 Skill 只负责判断“需要什么画布、画布表达什么”。保存和版本管理进入 `dimens-manager/references/canvas/overview.md` 或 `dimens-manager/references/canvas/overview.md`。

### 4.8.2 审批工作流画布

审批系统或审批流程需求必须同时拆两件事：

1. 审批工作流画布：表达发起、分支、审批、拒绝、撤回、通知、摘要回写。
2. 可执行审批工作流：进入 `dimens-manager/references/workflow/references/approval-generation.md`，生成 `pluginType=approval` 的工作流草案和项目落地计划。

不能只生成审批画布就声称审批能力完成。

### 4.9 报表与看板

至少要拆出：

- 谁看报表
- 报表回答什么问题
- 统计口径基于哪些表
- 是否需要参数筛选、联动、导出

不要把报表当成最后补的展示页。很多系统从一开始就要围绕“管理者看什么、执行者看什么、客户看什么”来决定表结构和权限边界。

### 4.10 外部接入

至少要判断：

- 是否需要 API Key / Secret
- 是否需要第三方脚本、开放接口、外部系统同步
- 是否只需要登录鉴权，还是还要开放业务接口
- 是否存在“外部系统可查但不可改”的边界

## 5. 默认输出模板

当用户只说“生成一个 XX 系统”时，默认应先输出这份主线模板：

```md
### 1. 系统定位
- 这是一个什么系统
- 主要服务哪些角色

### 2. 团队与项目上下文
- 团队 / 项目归属
- 是否存在多团队或多项目隔离

### 3. 核心业务对象
- 主对象
- 从对象
- 关键关系

### 4. 项目与表
- 项目名称
- 主表
- 从表
- 可选文档

### 5. 字段与关联
- 主展示字段
- 状态字段
- 时间字段
- 关联字段

### 6. 示例数据
- 每张表的案例数据

### 7. 查询案例
- 搜索
- 筛选
- 排序
- 关联查询

### 8. 可选扩展
- 权限
- 业务场景画布
- 工作流 / 审批工作流画布
- 报表
- 对接

### 9. 验收证据
- 项目与封面：project info
- 菜单归位：sheet tree
- 表结构：sheet info / column list / view list
- 示例数据：row page
- 文档：doc info
- 报表：report query
- 画布：canvas info / canvas save
```

这一步的目标是先让用户看到“系统至少要落到什么粒度”，而不是替用户预设 CRM、项目管理、售后或审批模板。

## 6. CRM 作为案例示例

典型模块：

- 客户表
- 联系人表
- 跟进记录表
- 商机表
- 销售阶段视图
- 销售审批流
- 商机推进业务场景画布
- 合同审批工作流画布
- 客户漏斗报表
- 外部系统同步能力

### 6.1 推荐拆解结果

| 维度 | 推荐输出 |
| --- | --- |
| 团队与项目上下文 | 销售团队、销售主管、销售成员、管理层；需要明确是团队级 CRM 还是某个项目下的客户项目 |
| 核心业务对象 | 客户、联系人、跟进记录、商机、合同、回款 |
| 角色与权限 | 销售只能看自己客户，主管看部门客户，管理层看全局；财务可看合同和回款但不能改客户基础资料 |
| 流程与自动化 | 商机推进、报价审批、合同审批、超期未跟进提醒、流失客户回访 |
| 业务场景画布 | 商机阶段流转画布、合同审批工作流画布、客户跟进异常路径画布 |
| 报表与看板 | 客户漏斗、销售转化率、阶段停留时长、销售排行、回款汇总 |
| 外部接入 | 外呼系统同步、官网线索导入、API Key 对接外部 CRM/ERP |

### 6.2 推荐优先路由

1. `dimens-manager/references/team/overview.md`
2. `dimens-manager/references/table/overview.md`
3. `dimens-manager/references/permission/overview.md`
4. `dimens-manager/references/workflow/overview.md`
5. `dimens-manager/references/canvas/overview.md`
6. `dimens-manager/references/canvas/overview.md`
7. `dimens-manager/references/report/overview.md`
8. `dimens-manager/references/key-auth/overview.md`

## 7. 不要直接省略的项

- 团队/项目隔离
- 字段设计
- 关联关系
- 示例数据
- 查询口径

这些如果不先拆清，后面会反复返工。

## 8. 总控 Skill 的最小闭环

一个合格的系统级输出，至少要形成下面闭环：

1. 先拆出模块。
2. 再给`dimens-manager` 章节 路由。
3. 再给接口导航。
4. 再标记哪些能力 `已封装`、哪些 `server-only`、哪些 `部分对齐`。
5. 如果进入执行阶段，最后必须给验证证据：上传 URL 是否写回、目录资源是否归位、表格字段/视图是否可查、报表预检链是否跑通。
6. 如果包含业务场景画布，必须说明画布 JSON 草案、保存链路和可执行工作流边界。

如果缺最后两步，后续 AI 很容易继续用抽象描述代替真实接口。

执行完成的最低验证口径：

| 资源 | 最低验证 |
| --- | --- |
| 项目 | `project info` 能看到名称、描述、封面/图标等更新结果 |
| 目录 | `sheet tree` 能看到表格、文档、报表处在目标目录下 |
| 表格 | `sheet info`、`column list`、`view list` 能回查到结构 |
| 数据 | `row page` 能查到示例数据，筛选/排序字段可用 |
| 报表 | `report preview`、`query-widget` 或 `query` 有结果 |
| 上传素材 | 上传返回 `url`，并已写回项目、文档或目标业务字段 |
